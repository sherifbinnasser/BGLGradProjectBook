\section{Overview}
The ultimate objective of this graduation project is to bridge the gap between academic deep learning research and practical, edge-based clinical utility. This chapter describes the implementation, software architecture, and production deployment of our non-invasive metabolic estimation system, officially named \textbf{GlucoAI}. 

Moving beyond isolated offline simulations, the entire system has been engineered into a fully functional, production-ready client-server architecture. The frontend application provides a modern, type-safe interface that allows users to capture high-density physiological signals on consumer-grade mobile hardware. These signals are then streamed to a high-performance remote inference engine that executes multi-task deep learning architectures to provide instantaneous, painless clinical predictions.

Unlike early-stage prototypes constrained by localized hardware bottlenecks, the production infrastructure is divided into two distinct components operating via an asynchronous network protocol:

\begin{enumerate}

It ingests client payloads, handles complex mathematical signal processing (Principal Component Analysis, 4th-order digital Butterworth filters, and cubic spline resampling), and feeds the standardized tensors into concurrent neural network pathways.
\end{enumerate}

\section{Image Analysis Framework}
To extract non-invasive physiological indicators from an optical source, the software configuration must access uncompressed camera sensor frames without adding computational lag. The mobile client implements an architecture centered around the Android CameraX `ImageAnalysis.Analyzer` infrastructure, allowing low-level access to individual pixel buffers directly from the hardware pipeline.

Camera sensors natively deliver frames utilizing the \textbf{YUV\_420\_888} color space standard, which decouples brightness information from color data to match human visual perception. This format distributes data across three distinct computational planes:
\begin{itemize}
    \item \textbf{Y Plane (Luminance):} Contains the absolute grayscale brightness values for every pixel, operating at the full native resolution of the selected camera hardware.
    \item \textbf{U Plane (Chroma Blue):} Represents the blue-difference color configurations, downsampled by half in both vertical and horizontal dimensions to minimize memory consumption.
    \item \textbf{V Plane (Chroma Red):} Represents the red-difference color configurations, downsampled identically to the U plane ($4:2:0$ chroma subsampling profile).
\end{itemize}

A primary technical challenge when dealing with hardware-level memory buffers is the varying data configurations across multiple Android chipsets. Sensor planes are bounded by explicit layout parameters known as the `rowStride` (the distance between rows of pixels in memory, which often includes padding) and the `pixelStride` (the distance between adjacent pixels). To ensure absolute execution stability and eliminate memory leakage or access violations across diverse smartphones, our analyzer pipeline reconstructs spatial configurations dynamically for every incoming frame, iterating through the memory layout without relying on predefined hardware assumptions.

To eliminate human contact transients, micro-motion artifacts, and sensor exposure adaptation lags that naturally occur when a finger first presses against a lens, the mobile software implements a \textbf{5-second automated initial trimming window}. The system records for a total of 25 seconds but drops the first 5 seconds of data, ensuring that only the remaining 20 seconds of highly stabilized biological signals are preserved for subsequent AI processing.

To maintain a flawless 30 FPS acquisition speed on lower-end mobile processors, the extraction engine utilizes a full-frame spatial integration strategy within a dedicated class named \texttt{FingertipAnalyzer}. Instead of executing traditional spatial subsampling which risks throwing away critical high-frequency cardiac pulse metrics, this pipeline ingests 100\% of the available pixel values inside the Y, U, and V memory planes. By calculating the absolute mathematical average intensity for each plane independently per frame, the engine acts as an extensive spatial low-pass filter, suppressing hardware static and maximizing the overall Signal-to-Noise Ratio (SNR).

import androidx.camera.core.ImageAnalysis
import androidx.camera.core.ImageProxy

/**
 * Analyzes camera frames to extract the average RGB intensity for PPG.
 * Processes every pixel of the frame (no subsampling) for maximum signal integration.
 */
class FingertipAnalyzer(
    private val onSignalExtracted: (y: Float, u: Float, v: Float, width: Int, height: Int) -> Unit
) : ImageAnalysis.Analyzer {

    override fun analyze(image: ImageProxy) {
        val width = image.width
        val height = image.height
        
        // Extract averages with proper pixel-stride handling across the full frame
        val yMean = calculateAverage(image.planes[0], width, height)
        val uMean = calculateAverage(image.planes[1], width / 2, height / 2)
        val vMean = calculateAverage(image.planes[2], width / 2, height / 2)

        onSignalExtracted(yMean, uMean, vMean, width, height)

        image.close()
    }

    private fun calculateAverage(plane: ImageProxy.PlaneProxy, width: Int, height: Int): Float {
        val buffer = plane.buffer
        val rowStride = plane.rowStride
        val pixelStride = plane.pixelStride
        
        var sum = 0L
        var count = 0
        
        buffer.rewind()
        for (y in 0 until height) {
            val rowOffset = y * rowStride
            for (x in 0 until width) {
                val index = rowOffset + x * pixelStride
                if (index < buffer.capacity()) {
                    sum += buffer.get(index).toInt() and 0xFF
                    count++
                }
            }
        }
        
        return if (count > 0) sum.toFloat() / count else 0f
    }
}

\begin{equation}
\text{Schema} = \{t, r, g, b\}

\section{Mathematical Signal Integration}
The statistical integration loop inside the analyzer layer is designed to handle hardware-level memory offsets natively. For a given frame plane with an active horizontal pixel domain $W$ and vertical domain $H$, the byte index $I(x, y)$ inside the unswapped 1D memory array is modeled continuously via the stride offset equation:
\begin{equation}
I(x, y) = (y \times \text{rowStride}) + (x \times \text{pixelStride})
\end{equation}
By executing a bitwise conjunction (\texttt{and 0xFF}) on the retrieved byte stream, signed Java integer primitives are mapped to true unsigned intensity coordinates bounded between $0 \to 255$.

The absolute spatial average component intensity $\mu_{\text{plane}}$ for that specific image proxy segment is calculated directly as:
\begin{equation}
\mu_{\text{plane}} = \frac{1}{\mathcal{N}} \sum_{y=0}^{H-1} \sum_{x=0}^{W-1} \mathcal{B}\left(\min\left(I(x, y), \text{capacity} - 1\right)\right)
\end{equation}
where $\mathcal{B}(i)$ indicates the mapped byte scalar value at index $i$, and $\mathcal{N}$ represents the global count of evaluated pixels within the safe bounds of the plane capacity. This strict algorithmic verification eliminates out-of-bounds pointer collisions, guaranteeing uniform cross-platform numerical stability.

\section{Concurrency Management}
Executing high-density frame ingestion requires strict asynchronous scheduling to prevent thread starvation on consumer-grade mobile systems. The GlucoAI client application isolates the hardware-intensive acquisition pipeline from the critical user interface thread through a decoupled multi-threaded architecture orchestrated via Kotlin Coroutines.

The primary image analysis workflow is allocated to a specialized thread context utilizing `Dispatchers.Default` for intensive computation, combined with a custom backpressure-aware production worker. By wrapping the CameraX framework inside an explicit production scope bound to the Android Lifecycle, memory allocations are automatically recycled during state transitions. 

Data values gathered from the spatial integration pipeline are continuously funneled into an asynchronous cold state manager (`MutableStateFlow`). This structure handles the live monitoring stream by applying a reactive throttle that downsamples the preview update frequency to 15Hz. This deliberate downsampling guarantees that the visual rendering layer remains smooth and operational without conflicting with the dedicated 30 FPS raw camera buffer consumption.

\section{Presentation State Management}
To bridge the gap between low-level hardware ingestion and declarative Jetpack Compose UI states, the application implements a centralized state architectural pattern managed via an Android architecture component `ViewModel`. The `MeasurementViewModel` acts as the single source of truth for the entire acquisition lifecycle, exposing strictly structured, read-only observable states to prevent unintended external side-effects.

A critical design consideration within the presentation layer is the optimization of spatial storage during the acquisition window. Instead of forcing heavy color space matrix transformations inside the live frame loop, the framework implements a deferred conversion mechanism. During the active logging window, the engine acts as an ingestion buffer, appending raw spatial averages directly into an internal allocation queue (`inferenceBuffer`). The execution of the resource-heavy YUV-to-RGB plane transformations and the execution of the 5-second initial trimming window are completely deferred to a background thread pooled via `Dispatchers.Default` once the recording sequence completes.

class MeasurementViewModel(

        private set

    var availableFps = mutableStateListOf<Int>()

    }

    fun updateSelectedResolution(size: Size?) {
        selectedResolution = size
    }

    private fun getFriendlyResolution(width: Int, height: Int): String {
        val minDim = minOf(width, height)
        val maxDim = maxOf(width, height)
        
        return when {
            minDim >= 2160 || maxDim >= 3840 -> "4K"
            minDim >= 1440 || maxDim >= 2560 -> "2K"
            minDim >= 1080 || maxDim >= 1920 -> "1080p"
            minDim >= 720 || maxDim >= 1280 -> "720p"
            minDim >= 480 || maxDim >= 640 -> "480p"
            else -> "${width}x${height}"
        }

        private set
    var liveG by mutableFloatStateOf(0f)
        private set
    var liveB by mutableFloatStateOf(0f)
        private set

    val liveRHistory = mutableStateListOf<Float>()
    val liveGHistory = mutableStateListOf<Float>()
    val liveBHistory = mutableStateListOf<Float>()
    private val maxWaveformPoints = 100

    private fun updateWaveformHistory(r: Float, g: Float, b: Float) {
        liveRHistory.add(r)
        if (liveRHistory.size > maxWaveformPoints) liveRHistory.removeAt(0)
        liveGHistory.add(g)
        if (liveGHistory.size > maxWaveformPoints) liveGHistory.removeAt(0)
        liveBHistory.add(b)
        if (liveBHistory.size > maxWaveformPoints) liveBHistory.removeAt(0)

        cameraResolution = getFriendlyResolution(width, height)

        liveG = (y - 0.344136f * (u - 128f) - 0.714136f * (v - 128f)).coerceIn(0f, 255f)
        liveB = (y + 1.772f * (u - 128f)).coerceIn(0f, 255f)
        updateWaveformHistory(liveR, liveG, liveB)

        isSignalStable = true

        if (isRecording) {
            inferenceBuffer.add(Triple(y, u, v))
        }
    }

    fun startRecording() {
        if (isRecording) return
        isRecording = true
        recordingProgress = 0f
        inferenceBuffer.clear()
        estimatedGlucose = null
        isProcessingComplete = false
        isInferenceRunning = false

        viewModelScope.launch {
            val totalDurationMs = 25000L
            val stepMs = 100L
            var elapsedMs = 0L
            
            while (elapsedMs < totalDurationMs && isRecording) {
                delay(stepMs)
                elapsedMs += stepMs
                recordingProgress = elapsedMs.toFloat() / totalDurationMs
            }
            if (isRecording) {
                stopRecording()
            }
        }
    }

    fun stopRecording() {
        if (!isRecording) return
        isRecording = false
        if (!isInferenceRunning && !isProcessingComplete && inferenceBuffer.isNotEmpty()) {
            runInference()
        }
    }

    private fun runInference() {
        if (isInferenceRunning || isProcessingComplete) return
        if (inferenceBuffer.isEmpty()) return
        
        isInferenceRunning = true
        isProcessingComplete = false
        isRecording = false
        
        val currentFps = selectedFps ?: 30
        val rawSamples = inferenceBuffer.toList()
        inferenceBuffer.clear()

        viewModelScope.launch {
            val rgbSamples = withContext(Dispatchers.Default) {
                val samplesToDrop = currentFps * 5
                rawSamples.drop(samplesToDrop).map { (y, u, v) ->
                    val r = (y + 1.402f * (v - 128f)).coerceIn(0f, 255f)
                    val g = (y - 0.344136f * (u - 128f) - 0.714136f * (v - 128f)).coerceIn(0f, 255f)
                    val b = (y + 1.772f * (u - 128f)).coerceIn(0f, 255f)
                    Triple(r, g, b)
                }
            }

            delay(1000)

            val response = withContext(Dispatchers.Default) {
                glucoseRepository.estimateGlucoseFromApi(rgbSamples, currentFps)
            }
            
            estimatedGlucose = response.glucoseValue
            glucoseClass = response.glucoseClass
            bloodPressure = response.bloodPressure
            isInferenceRunning = false
            isProcessingComplete = true
        }
    }

    fun resetMeasurement() {
        estimatedGlucose = null
        glucoseClass = null
        bloodPressure = null
        isInferenceRunning = false
        isProcessingComplete = false
        isRecording = false
        recordingProgress = 0f
        inferenceBuffer.clear()
        liveRHistory.clear()
        liveGHistory.clear()
        liveBHistory.clear()
    }
}

                isLenient = true
            })
        }

        ppgSignal: List<Triple<Float, Float, Float>>,
        fps: Int = 30
    ): PredictionResponse {
        return try {
            val csvContent = StringBuilder()
            csvContent.append("t,r,g,b\n")
            
            val dt = 1.0f / fps
            ppgSignal.forEachIndexed { i, sample ->
                val t = i * dt
                csvContent.append("$t,${sample.first},${sample.second},${sample.third}\n")
            }

            val url = "https://abdelrahmanrabie-graduation-project.hf.space/predict/all"

            val response: PredictionResponse = client.post(url) {
                setBody(MultiPartFormDataContent(
                    formData {
                        append("file", csvContent.toString().toByteArray(), Headers.build {
                            append(HttpHeaders.ContentType, "application/octet-stream")
                            append(HttpHeaders.ContentDisposition, "form-data; name=\"file\"; filename=\"data.csv\"")
                        })
                    }
                ))
            }.body()

            Log.d("GlucoseRepository", "API Response: $response")
            response
        } catch (e: Exception) {
            Log.e("GlucoseRepository", "API request failed", e)
            PredictionResponse(glucose = -1f, bloodPressure = BloodPressure(0f, 0f))
        }
    }

    fun close() {
        client.close()
    }

\section{Backend Infrastructure}
Upon receiving the payload at the `/predict/all` endpoint via the client data layer, the FastAPI backend routes the raw time-series matrices through an advanced Python preprocessing and execution pipeline. The backend architecture performs four crucial operations:
\begin{enumerate}
    \item \textbf{Dynamic Sampling Rate Computation:} Computes the median timestamp delta ($\Delta t$) to dynamically calculate the actual hardware sampling frequency ($f_s = 1 / \Delta t$).
    \item \textbf{Principal Component Analysis (PCA):} Projects the three-channel RGB arrays into orthogonal component spaces, automatically selecting the dominant physiological vector showcasing the highest variance.
    \item \textbf{Digital Signal Filtering and Resampling:} Applies a 4th-order digital Butterworth bandpass filter to restrict frequencies between $0.5\text{ Hz}$ and $1.5\text{ Hz}$ (correlating to human heart rates between 30 and 90 BPM), followed by a cubic spline interpolation that standardizes the signal array to exactly $400$ input dimensions.
    \item \textbf{Concurrent Deep Learning Inference:} Executes three specialized models simultaneously: a Blood Pressure CNN network, a fine-tuned deep regression Glucose Network (`RigidGlucoseNet`), and a Multi-Task classification model (`MultiTaskPPGModel`).
\end{enumerate}

def bandpass_filter(signal, fs, low=0.5, high=1.5, order=4):
    nyq = 0.5 * fs
    high = min(high, nyq * 0.99)
    b, a = butter(order, [low / nyq, high / nyq], btype="band")
    return filtfilt(b, a, signal)


def extract_pca_signal(rgb, fs):
    """Extract PPG signal using PCA + select best component."""
    rgb = rgb.astype(np.float32)
    rgb = (rgb - np.mean(rgb, axis=0)) / (np.std(rgb, axis=0) + 1e-8)

    pca = PCA(n_components=3)
    components = pca.fit_transform(rgb)

    best_signal = None
    best_score = -1

    for i in range(2):
        sig = components[:, i]
        try:
            sig_f = bandpass_filter(sig, fs)
            score = np.std(sig_f)
            if score > best_score:
                best_score = score
                best_signal = sig_f
        except Exception:
            continue

    if best_signal is None:
        best_signal = components[:, 0]

    return best_signal


def adaptive_resample(signal, target_len=FIXED_POINTS):
    n = len(signal)
    if n == target_len:
        return signal

    x_old = np.linspace(0, 1, n)
    x_new = np.linspace(0, 1, target_len)

    if n < target_len:
        interpolator = interp1d(x_old, signal, kind="cubic", fill_value="extrapolate")
        return interpolator(x_new)
    else:
        return resample(signal, target_len)

        if c not in df.columns:
            raise HTTPException(
                status_code=400,
                detail="CSV must contain columns: t,r,g,b"
            )

    rgb = df[["r", "g", "b"]].values.astype(np.float32)

    time = df["t"].values.astype(np.float64)
    deltas = np.diff(time)
    deltas = deltas[(deltas > 0)]
    dt = np.median(deltas) / 1000.0
    fs = 1.0 / dt

    ppg = extract_pca_signal(rgb, fs)

    ppg = (ppg - np.mean(ppg)) / (np.std(ppg) + 1e-8)
    ppg = np.clip(ppg, -3, 3)

    ppg = adaptive_resample(ppg, FIXED_POINTS)

    tensor = torch.tensor(ppg, dtype=torch.float32)
    tensor = tensor.unsqueeze(0).unsqueeze(0)  # (1, 1, 400)

    return tensor.to(device)

\section{Filtering Metrics}
The raw temporal sequence received by the cloud backend is natively unconditioned and heavily corrupted by high-frequency white noise and low-frequency baseline wander caused by minor respiration shifts. To address this, the Python execution layer implements a strict digital filtering pipeline before vector projection.

The 4th-order digital Butterworth filter is mathematically formulated to calculate stabilized downstream arrays by executing a zero-phase digital filtering operation (`filtfilt`). This technique filters the input data in both the forward and reverse directions to eliminate phase distortion completely, which is vital for preserving the delicate peak locations of the pulse waves. The discrete transfer function is defined as:
\begin{equation}
H(z) = \frac{\sum_{i=0}^{M} b_i z^{-i}}{\sum_{j=0}^{N} a_j z^{-j}}
\end{equation}
where $a$ and $b$ represent the filter coefficients dynamically generated via `scipy.signal.butter` relative to the computed hardware sampling frequency ($f_s$).

Following filtering, the signal undergoes adaptive cubic spline interpolation. This operation fits a series of piecewise cubic polynomials across the irregular temporal points. This process ensures that regardless of whether the mobile hardware collected 580 or 620 raw frames during the operational window, the tensor injected into the neural pathways is mathematically bounded to exactly $400$ dimensions, satisfying the input requirements of the multi-task model architecture.

\section{UI/UX Failure Recovery}
Deploying a diagnostic system on diverse consumer hardware exposes the processing pipeline to extreme environmental variances. The clinical presentation layer embedded within GlucoAI incorporates protective software exceptions designed to capture, isolate, and handle acquisition errors.

If the internal preprocessing pipeline detects that the computed hardware frequency falls below an acceptable threshold ($f_s < 15\text{ FPS}$), the server rejects the instance and returns an explicit `HTTP 400` error to the mobile client. Similarly, the PCA projection module evaluates the signal variance across the secondary components; if the maximum standard deviation drops below a preset clinical threshold ($1\times10^{-5}$), the system flags the capture as invalid due to poor contact or insufficient transillumination.

The frontend client intercepts these network anomalies within a safe presentation wrapper, catching exceptions before they reach the main UI context. Instead of presenting uncalibrated or unstable estimations, the application updates the UI state to display an intuitive error recovery card. This module provides actionable feedback, prompting the user to adjust their sensor placement or relocate to an environment with better ambient lighting before re-initializing the diagnostic loop.
